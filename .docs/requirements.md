# プロダクト要件定義・概要設計書：Irori（イロリ）

## 1. プロダクト概要
### 1.1 コンセプト
「遠くの灯りを眺めるように、シーンに合わせて『ノリ』と『文脈』をチューニングして集まる、囲炉裏型ライフスタンス・チャット」

### 1.2 背景と目的
既存のチャットツールの「常に繋がっている」「機能が多すぎる」状態から生じる既読疲れや心理的コストを解消。
システムに人間が合わせるのではなく、ユーザー関係性に合わせて通信の深度や機能（既読、タイピング表示、スレッド等）を柔軟にコントロールできる居心地の良いデジタル空間を提供する。

---

## 2. システム要件（機能要件）
### 2.1 アカウント・フレンド機能
- **Googleログイン（OAuth認証）**: Google Picker APIとのシームレスな連携（アクセストークンの共有）を前提とし、初期フェーズではGoogleによるパスワードレスな認証のみをサポートする（自前のメール認証や他社OAuthは持たない）。
- **フレンド機能**: 招待リンクやQRコードを利用した1対1のフレンド登録。
- **ステータス（気配）表示**: プロフィール横に限定的な数種類（「ひま」「作業中」など）のランプのみを灯す。

### 2.2 コミュニケーション機能（ルーム内）
- **視覚的UI**: 1つの会話の塊が直感的に識別できるタイムライン。
- **リアクション**: 厳選された絵文字（🔥、🍵など）によるワンタップリアクション。選ぶ疲れを排除する。カスタムリアクションについても提供。
- **メッセージ操作**: 時間制限のない事後編集および送信取り消し（完全削除）。
- **未読戻し**: 開封済みルームを意図的に「未読」に戻し、返信タスクをマイペースに管理する機能。
- **引用（リプライ）**: 過去ログを背景にした返信機能。

### 2.3 ゼロ・ストレージメディア連携
- **Google Picker API連携**: 自前ストレージを持たない。Googleログインから得られた認証トークンを利用し、ユーザー自身のGoogle Drive/Photosの画像をURLとしてチャットに埋め込む。
- **OGP展開**: URLリンクからアプリ内で自動的にサムネイルを展開表示。インフラへの負荷・法的リスクを極小化する。Google Drive/Photosの画像・動画の場合はAPIからソースリンクを生成し、チャット画面上でプレビュー可能にする。

### 2.4 ルーム設定（トグル制御）
ルーム単位で管理者が以下の機能をON/OFF可能：
1. **既読の可視化**: 閲覧時に既読をつけるか否か。
2. **タイピング表示**: 相手の入力ステータスをリアルタイム表示するか否か。
3. **スレッド機能**: 話題を分岐させる独立タイムラインを許可するか。
4. **メンション機能**: 「＠ユーザー名」の強制通知を許可するか。
5. **一時チャットモード**: 24時間でメッセージや画像を自動消去するか否か。

### 2.5 外部連携 (Inbound API)
- 外部システム（GitHubやタスク管理等）から認証トークンを用いて、特定ルームへ自動投稿するWebhook機能。流し読み専用ルームとして活用可能。

---

## 3. 非機能要件
- **コスト・インフラ最適化**: 個人開発の無料枠（Cloudflareスタック等）を最大限活用し、固定費ゼロを死守するシステム設計。
- **法的・運用リスク極小化**: クローズドな招待制とゼロ・ストレージ戦略により、違法メディアのホスティング等の法的リスクをサードパーティ（Google等）へ委譲する。
- **パフォーマンス**: Hono + Cloudflare Workersのエッジ環境による低遅延API、WebSocketによるリアルタイム通信。

---

## 4. 画面・UI設計概要（ダッシュボード）
ダークテーマベースの3カラム構成：

- **サイドバー（左）**:
  - ユーザー自身のステータスランプ切り替え。
  - 参加中のルーム（囲炉裏）一覧と未読バッジ。
  - フレンド追加・招待アクションへの導線。
- **メインチャットエリア（中央）**:
  - メッセージのタイムライン表示（スクロール遡り）。
  - APIによるGoogle Drive/Photosのインライン画像表示。
  - メッセージ入力フォーム（Google Picker呼び出しアイコン配置）。
- **設定パネル（モーダルメニュー）**:
  - ルームの「ノリ」を決める、既読、タイピング、スレッド、メンション、一時チャットモードのトグルスイッチ群。

## 5. 概要設計 (High-Level Design)
### 5.1 システムアーキテクチャ
モノレポ（pnpm workspaces）構成を採用。
- **フロントエンド**: Vercel + Next.js (App Router), Tailwind CSS v4, shadcn/ui, base-ui. 
- **バックエンド**: Cloudflare Workers + Hono.
- **データベース**: Cloudflare D1 (SQLite).
- **リアルタイム通信**: Cloudflare Durable Objects / Cloudflare Workers WebSocket.

### 5.2 API・リアルタイム通信アーキテクチャ
Honoの **Type-safe RPC** を利用し、フロント・バック間のエンドツーエンドの型安全を実現。

#### HTTP REST エンドポイント (`/api/v1/*`)
状態変更やデータ取得、メッセージの投稿・編集・取り消しのトリガーに利用。

#### WebSocket エンドポイント (`/ws/rooms/:roomId`)
リアルタイム通信専用の独立パス。サーバー・クライアント間でJSON文字列をパースして相互通信する。

##### サーバー ➡️ クライアント（ブロードキャストイベント）
- `NEW_MESSAGE`: 新規メッセージが投稿された際、ルーム内の全員のUIに即時反映。
- `MESSAGE_EDIT`: HTTP側でメッセージの事後編集が成功した際、該当メッセージの内容を全員のUIでリアルタイムに書き換える。
- `MESSAGE_DELETE`: HTTP側で送信取り消し（削除）が成功した際、該当メッセージのUIを全員の画面から即座に消去する。
- `USER_TYPING`: タイピング状態（開始・終了）の変化をDBを介さずに高速ブロードキャスト。

### 5.3 主要データモデル・永続化方針
- **User**: `id`, `name`, `google_oauth_id`, `status_lamp`
- **FriendConnection**: `user_id_a`, `user_id_b`, `created_at`
- **Room**: `id`, `name`, `created_at`
- **RoomMember**: `room_id`, `user_id`, `role`, `joined_at`
- **RoomSettings**: `room_id`, `is_read_visible`, `is_typing_visible`, `is_thread_enabled`, `is_mention_enabled`, `is_ephemeral_mode`
- **Message**: `id`, `room_id`, `user_id`, `content`, `media_url`, `reply_to_id`, `created_at`, `updated_at`

#### 一時チャット（24時間消滅）の実装方針
サーバーレス環境（Cloudflare Workers）の制限に合わせ、インフラ負荷と実装コストを最小化する**フィルター駆動型消滅**を採用する。
1. **画面上の消滅（クエリフィルター）**: メッセージ取得（GET API）のSQLクエリ側で、ルームの `is_ephemeral_mode` がONの場合、`WHERE created_at > datetime('now', '-24 hours')` 条件を付与し、24時間が経過したメッセージを画面上から即座に不可視（消滅）にする。
2. **物理削除（非同期クローン）**: 物理的なDBからのデータ削除は、Cloudflare Triggers (Cron) を用いて1時間に1回等の定時バッチを実行し、24時間を超えた不要データを一括削除（`DELETE`）する。

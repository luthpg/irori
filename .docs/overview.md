# プロダクト概要書：Irori（イロリ）

## 1. 企画の骨子
Irori は、親しい人同士が「火の周りにふらっと集まる」ような安心感を持てる、ルーム単位で招待制を担保できるチャットです。アプリアカウント自体は任意で作成可能にし、参加するルームごとに既読・タイピング表示・メンション・一時チャットといった通信の距離感を調整できることが中核価値です。

### 1.1 コンセプト
「遠くの灯りを眺めるように、シーンに合わせて『ノリ』と『文脈』をチューニングして集まる、招待制ルーム型の囲炉裏チャット」

### 1.2 背景
既存のチャットは「常につながっている」「通知が多い」「履歴が残りすぎる」ことで、気軽な会話の摩擦が増えやすいです。Irori は、ルーム単位でその距離感を設計できる閉じた空間として提供します。

### 1.3 提供価値
- ルームごとに既読・タイピング表示・スレッド・メンション・一時チャットを切り替えられる
- ルーム単位で招待制を設定でき、アカウントは任意作成とすることで参加ハードルを抑える
- Google Drive / Photos 由来の URL を扱い、アプリ側のストレージ負荷を最小化する
- 初期フェーズでは「軽さ」と「実装容易さ」を優先し、MVP をスリムに構築する

## 2. 競合との比較

### 2.1 LINE
- 強み: 国内インフラとして圧倒的な認知
- 弱み: 既読圧力や通知過多が強い
- Irori の差別化: ルーム単位で「距離感」を制御できる

### 2.2 Discord
- 強み: 高機能でコミュニティ運営に強い
- 弱み: 日常会話には複雑で情報過多になりやすい
- Irori の差別化: 小さな親密な空間に集中した、軽量で心理的に安全な体験

### 2.3 Irori
- 強み: 招待制・ルーム別設定・ゼロストレージ寄りのメディア共有
- 弱み: 初期はネットワーク効果が弱い
- 機会: 通知疲れや情報過多に疲れたユーザー層

## 3. MVP の機能スコープ

### 3.0 技術前提
- フロントエンド: Next.js (App Router) + Tailwind CSS v4 + shadcn/ui + @base-ui/react
- バックエンド: Hono + Cloudflare Workers
- データベース: Cloudflare D1
- リアルタイム: WebSocket / Durable Objects
- 命名規約: API/フロントエンドは camelCase、DB は snake_case

### 3.1 アカウント・フレンド
1. Google ログイン
2. フレンド登録（招待リンク / QRコード / 1対1招待）
3. 気配表示（「ひま」「作業中」「離席」などの限定的なステータスランプ、内部値: `free` | `busy` | `away`）

### 3.2 ルーム内コミュニケーション
1. ルーム単位のタイムライン
2. リアクション（限定セットの絵文字）
3. メッセージ編集・削除
4. 未読戻し
5. 引用（リプライ）

### 3.3 メディア共有
1. Google Picker API を使った画像選択
2. Google Drive / Photos の共有 URL をチャットに埋め込む
3. アプリ側ではバイナリを保存せず URL 文字列を扱う

### 3.4 ルーム設定
各ルームは招待制として運用でき、管理者は以下を ON / OFF できる。
1. 既読の可視化
2. タイピング表示
3. スレッド
4. メンション
5. 一時チャットモード（24 時間で自動消去）

### 3.5 外部連携
1. Inbound Webhook / REST API によるメッセージ投稿
2. 一時チャットモードのルームで bot 通知を流し読み向けに活用

> 初期リリースでは、外向き通知や大規模な公開チャンネルは対象外とする。

---

## 4. 品質と運用の方針
- 個人開発のコストを抑えるため、Cloudflare スタックとモノレポ構成を前提にする
- 法的・運用リスクを低減するため、メディアの実体は Google 側に寄せる
- まずは「軽さ」「安全性」「実装の再現性」を優先し、機能追加は段階的に行う

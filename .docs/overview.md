# プロダクト概要書：Irori（イロリ）

## 1. プロダクトコンセプト
**「遠くの灯りを眺めるように、シーンに合わせて『ノリ』と『文脈』をチューニングして集まる、完全招待制の囲炉裏型ライフスタンス・チャット」**

* **背景**: 既存のチャットツールは「常に繋がっている」「機能が多すぎる」など、システム側にコミュニケーションの距離感が固定されており、既読疲れや通知の圧迫、情報過多による心理的コストを生み出している。
* **提供価値**: キャンプの焚き火やホテルのラウンジのように、親しい人たちと火の周りにふらっと集まるような居心地の良さをデジタル上に再現。全ルームを「完全招待制」のクローズドな空間とし、部屋ごとに既読、タイピング表示、スレッド、メンション、履歴の自動消去（一時チャット）をワンタップでコントロール可能。さらにGoogle Drive/Photosとシームレスに連携し、自社サーバーに重いデータを置かない「ゼロ・ストレージアーキテクチャ」を実現する。

## 2. 競合サービスとのSWOT分析

### 2.1 LINE (国内インフラ型)
* **S (強み)**: 国内9,700万人以上の圧倒的シェア。全世代が使うインフラ。
* **W (弱み)**: プライベートも業務もすべて1つのタイムラインに混在し、既読プレッシャーが強い。
* **O (機会)**: 「LINE離れ」「SNS疲れ」層の増加。
* **T (脅威)**: スレッド機能や一時消去機能の模倣。

### 2.2 Discord (コミュニティ・ボイス型)
* **S (強み)**: 柔軟な権限管理、スレッド、豊富なAPI。
* **W (弱み)**: UIがギーク向けで複雑。日常のライトな連絡にはオーバースペックであり、情報過多になりやすい。
* **O (機会)**: 少人数コミュニティの細分化。
* **T (脅威)**: 大手プラットフォームによる軽量版アプリのリリース。

### 2.3 本アプリケーション：Irori（シーン適応型・引き算のチャット）
* **S (強み)**: 
    * 部屋ごとの「通信の深度（既読・タイピング）」と「機能（スレッド・メンション）」を柔軟にON/OFF可能。
    * 完全招待制による心理的安全性の高いクローズド空間。
    * Google Picker API等を活用した画像共有により、アプリ側のストレージコストを極小化。
    * 24時間で全履歴が消える「一時チャットモード」とGoogle APIのURL有効期限の仕様が完璧にシンクロ。
* **W (弱み)**: ネットワーク効果（ユーザー数）がゼロからのスタート。
* **O (機会)**: 既存の通知や機能の多さに疲弊した層の受け皿。Inbound APIを用いた、開発者向けの「流し読み専用ログルーム」の構築。
* **T (脅威)**: Google APIの仕様変更や利用制限。

## 3. 差別化マトリクス

| 既存アプリのペイン（課題） | 競合の限界 | Irori（本アプリ）の解決アプローチ |
| :--- | :--- | :--- |
| **既読・即レスのプレッシャー** | LINE：常に既読がつく | **「ルーム別既読ON/OFF」×「未読に戻す」**<br>重要連絡はON、雑談はOFF。後で返したいものはワンタップで未読に戻す。 |
| **呼び出しによる集中力の阻害** | Slack/Discord：メンションで強制通知 | **「メンションのON/OFF制御」**<br>部屋によっては「＠ユーザー名」の通知機能を完全無効化し、ふらっと立ち寄る非同期の空間を維持。 |
| **デジタルタトゥー・容量圧迫** | 一般的なチャット：画像や履歴が永続化 | **「Google連携」×「一時チャットモード」**<br>自前ストレージを持たずGoogleのURLのみを参照。一時モードなら24時間でURLごとチャットが消滅。 |
| **オープンすぎる空間の治安悪化** | X/オプチャ：スパムや荒らしの侵入 | **「完全招待制（クローズド空間）」**<br>フレンド機能を経由した招待URLのみで入室可能。心理的安全性が高く、運用監視コストも最小化。 |

## 4. 最終機能要件一覧

### 4.1 コア・アカウント＆フレンド機能
1.  **フレンド機能**: ユーザーID検索、招待リンク、QRコード読み取りによる1対1のフレンド登録。ルームへ招待するための前提となるハブ機能。
2.  **ステータス共有**: プロフィール横に「ひま」「作業中」など極めて限定的な2〜3種類のランプ（絵文字）だけを灯し、遠くの気配を演出する。

### 4.2 コア・コミュニケーション機能
1.  **視覚的に識別しやすいUI**: 1つの会話の塊が直感的に視覚識別できるデザイン。
2.  **リアクション機能**: 厳選された数種類（🔥、🍵など）の絵文字に制限し、選ぶ疲れを排除したワンタップリアクション。
3.  **事後編集＆送信取り消し**: 時間制限なしでメッセージの修正・完全削除が可能。
4.  **未読に戻す機能**: 開封済みのルームを意図的に「未読」に戻し、自分のペースで返信タスクを管理。
5.  **引用機能（リプライ）**: 過去のメッセージを背景に敷いて返信する基本機能。

### 4.3 ゼロ・ストレージ型のメディア共有（Google API連携）
1.  **Google Picker API連携**: 入力欄の添付アイコンからGoogleドライブのUI（Picker）を直接開き、ユーザーのDrive/Photos内の画像を選択してチャットに埋め込む機能。
2.  **リンクの自動プレビュー展開（OGP型）**: Google DriveやGoogle Photosの共有リンクをテキストで貼り付けると、アプリ内で自動的にサムネイル画像が展開される機能。
    * *※ メリット*: 画像実体はGoogle側にあり、IroriのDBにはURL文字列のみを保存するためインフラ負荷が実質ゼロ。

### 4.4 ルーム設定（完全招待制・機能のトグル制御）
すべてのルームは招待制であり、管理者は部屋の「ノリ」に合わせて以下の機能をON/OFFできる。

1.  **既読の可視化 (ON/OFF)**: 閲覧時に既読をつけるか否か。
2.  **タイピング中表示 (ON/OFF)**: 相手の入力ステータスをリアルタイム表示するか否か。
3.  **スレッド機能 (ON/OFF)**: 話題を分岐させる独立タイムラインを許可するか、引用のみの一本道にするか。
4.  **メンション機能 (ON/OFF)**: 「＠ユーザー名」による名指しの強制通知を許可するか否か。
5.  **一時チャットモード (ON/OFF)**: 送信後24時間が経過すると、メッセージ・引用・Google画像リンク等すべてが自動消去されるフロー型の空間にするか否か。

### 4.5 外部連携・Inbound API
1.  **Inbound Webhook / REST API**: 外部システム（GitHubやタスク管理ツール）から認証トークンを用いて、特定のルームへメッセージを自動投稿できる機能。
    * *※ Outbound（Iroriから外部への通知）は意図的に排除し、閉じた空間の安心感を担保。*
2.  **一時チャット型Botワークフロー**: API経由の通知であっても、ルームが「一時チャットモード」なら24時間で自動消滅。ログが無限に溜まらない、流し読み専用の綺麗な通知ルームを構築可能。
